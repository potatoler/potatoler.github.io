Tricks and Traps in OI

2020-05-22
  • 减法取膜前记得先加模数
  • #define int long long 谨慎使用：有时不开会 WA，有时开了会 MLE
  • 并查集中代表元素赋为 0 时可能会出错，在与其他数据结构或数据处理方式搭配时要分外注意
  • Init 方法一定要写在主方法里面
  • 中间输出和 freopen 一定要记得删，为了防止删除中间输出可以使用 cerr 进行中间输出，但是如果不删会使程序运行变慢
  • 位运算加括号
  • INT_MAX 并不是任何时候都可以作为极大值使用，998244353 * 3 > INT_MAX
  • 线性筛预处理素数时记清楚处理范围是 n 还是 MaxN
  • set.erase() 可以使用迭代器作为参数，删除迭代器指向的元素
  • puts() 输出结尾自动换行
  • 高斯消元先判无解再判无穷解
  • if 语句多个判断条件时先判断下标范围再判断下标所对应的值 —— 当下标不合法时访问数组会越界，这时候先判断数组会 RE，而如果吧下标写在前面，会先判断下标，下标判断为伪时会直接结束判断。这样就防止了访问到不合法的位置。例如 if(pd >= D[i].size() || C[i][pc].score > D[i][pd].score) 这个语句，如果两个条件反过来写，pd 已经超出 D[i].size() 的情况下仍然访问了 D[i][pd].score，就会 RE
  • 注意 n 和 m 的读入顺序
  • 注意编号从 0 开始还是从 1 开始
  • 注意不要重定义变量！有时候编译器不会报错，死都不知道怎么死的。可以添加编译选项 -Wshadow 让编译器抛出警告
  • 一种防止被卡常的优化：提前存访问序列。例如六个点的访问顺序是 132504，一般来讲可以写一个 nextStep[] 数组，即 nextStep[6] = {4, 3, 5, 2, -1, 0}，但是也可以把他们放在一个访问队列中：queue[6] = {1,3,2,5,0,4}，访问时遍历这个队列。两者理论复杂度是一样的，但是注意到前者是乱序访问 nextStep，后者是顺序访问 queue，在数据量很大时前者常数较大
  • 当 Edge 中有很多元素时一定要记得检查 AddEdge() 函数有没有写全
  • 基环树的性质：断掉环上的任意一条边变成一棵树。所以一些题数据结构暗示图的结构只可能是树或基环树，可以先考虑树的做法，暴力枚举断掉的边去做基环树的情况
  • 当问题规模很大时，如果问题在 0/1 范围内很简单，可以考虑将原问题按照二进制分位做，最后再合并得出答案
  • 关系可以抽象成边，从而将问题抽象在一个图上求解
